# Multi-property architecture

One owner, multiple properties. Each property has its own rooms/menu/bookings/website/staff. A property switcher in the sidebar lets the owner jump between them.

## Current state

The system treats `tenants` as both "the account" and "the property." There's a 1:1 relationship: one auth user → one tenant row. Every data table (rooms, reservations, menu_items, staff, etc.) uses `tenant_id` as FK.

**Problem:** A user who owns 3 hotels can only manage 1. They'd need 3 separate logins.

## Proposed data model

### Option: Keep `tenants` as properties, add a mapping table

This is the least-disruptive approach. The existing `tenant_id` FK chain stays intact. We add a join table linking users to tenants.

```sql
-- New table: links auth users to tenants (properties) they can access
CREATE TABLE user_properties (
    id            bigint GENERATED BY DEFAULT AS IDENTITY PRIMARY KEY,
    user_id       uuid NOT NULL REFERENCES auth.users(id) ON DELETE CASCADE,
    tenant_id     uuid NOT NULL REFERENCES tenants(id) ON DELETE CASCADE,
    role          text NOT NULL DEFAULT 'SUPER_ADMIN',  -- per-property role
    is_owner      boolean DEFAULT false,                -- true = created this property
    created_at    timestamptz DEFAULT now(),
    UNIQUE(user_id, tenant_id)
);

-- Enable RLS
ALTER TABLE user_properties ENABLE ROW LEVEL SECURITY;

-- Users can see their own rows
CREATE POLICY "users_own_properties" ON user_properties
    FOR ALL USING (auth.uid() = user_id);
```

### Migration path for existing data

```sql
-- Backfill: create user_properties rows from existing tenants.owner_id
INSERT INTO user_properties (user_id, tenant_id, role, is_owner)
SELECT owner_id, id, 'SUPER_ADMIN', true
FROM tenants
WHERE owner_id IS NOT NULL;
```

This is a zero-downtime migration. Existing data is preserved. The old `owner_id` column on `tenants` stays as a fallback.

### RLS update

Existing RLS policies check `tenant_id = (SELECT id FROM tenants WHERE owner_id = auth.uid())`. After this change, they should check:

```sql
tenant_id IN (SELECT tenant_id FROM user_properties WHERE user_id = auth.uid())
```

This automatically gives a user access to all their properties' data.

## Frontend changes

### 1. AuthContext.tsx

**Current:** Fetches a single tenant via `.eq('owner_id', userId).single()`.

**New:**
- Fetch all tenants the user has access to via `user_properties`
- Store them in a `properties: Tenant[]` array
- Track `activeTenant: Tenant` (the currently selected property)
- Persist the selection in `localStorage` (key: `active_property_id`)
- Expose `switchProperty(tenantId)` function
- The user's **role** comes from `user_properties.role` (per-property), not from `auth.users.user_metadata.role`

```typescript
interface AuthContextType {
    user: User | null;
    tenant: Tenant | null;           // the ACTIVE property
    properties: Tenant[];            // all properties this user can access
    session: Session | null;
    loading: boolean;
    login: (...) => Promise<...>;
    signup: (...) => Promise<...>;
    logout: () => Promise<void>;
    switchProperty: (tenantId: string) => void;
    addProperty: (name: string, type: BusinessType) => Promise<{ error: string | null }>;
    hasPermission: (allowedRoles: Role[]) => boolean;
    refreshTenant: () => Promise<void>;
}
```

### 2. Sidebar.tsx — property switcher

Add a dropdown at the top of the sidebar, above the nav items:

```
┌─────────────────────────┐
│  Ganesh Residency    ▾  │  ← dropdown trigger
│  Hotel                  │
├─────────────────────────┤
│  ✓ Ganesh Residency     │  ← active
│    Beach House          │
│    Mountain Lodge        │
│  ──────────────────     │
│  + Add property         │
└─────────────────────────┘
```

- Shows the current property name and type
- Dropdown lists all `properties`
- Checkmark on the active one
- "Add property" at the bottom opens a modal
- Switching triggers `switchProperty()` → reloads CRMDataContext

### 3. CRMDataContext.tsx

No structural changes needed. It already queries everything by `tenant.id`. When `switchProperty()` fires, the `tenant` in AuthContext changes, which triggers `useEffect` in CRMDataContext to re-fetch data for the new property.

### 4. Signup flow

**First-time signup** stays the same — creates a user + one tenant + one `user_properties` row.

**Adding a second property** (from the sidebar "Add property" button):
1. Creates a new `tenants` row (via RPC or direct insert)
2. Creates a `user_properties` row linking the user to it
3. Generates a new subdomain
4. Switches to the new property

### 5. Staff / role system

When an owner invites staff to a specific property:
1. Staff signs up (or gets invited via email)
2. A `user_properties` row is created with `role = 'RECEPTION'` and `tenant_id = <that property>`
3. The staff member only sees properties they're assigned to
4. Their role can differ per property (MANAGER at hotel A, RECEPTION at hotel B)

### 6. TenantSite.tsx (public website)

No changes needed. It already resolves by subdomain → tenant_id. Each property has its own subdomain, so the routing works as-is.

## Risk assessment

**Low risk:** The `tenant_id` FK structure doesn't change. All existing queries continue to work. The only change is *where the active tenant_id comes from* (AuthContext) and *who can access which tenant_id* (RLS).

**Medium risk:** RLS policy updates on 15 tables. If any policy is missed, that table becomes inaccessible. Mitigation: run a test query against each table after migration.

**Not affected:** TenantSite.tsx, LandingCMS, WebsiteEditor — these already work per-subdomain/tenant_id.

## Verification

1. Existing user logs in → sees their current property, works exactly as before
2. User adds a second property → appears in dropdown, can switch to it
3. Data is isolated — switching properties reloads all CRM data
4. Staff invited to property B can't see property A's data
5. Each property has its own subdomain and website
